# BurnRate v6 — Progress Log

Tracks each feature from `docs/specs/goal6.md` through Started → Decisions → Tests added → Verified.

## Feature 1 — Household profiles + cost splits — Lib complete

- **Started**: 2026-05-14
- **Decisions**: `src/lib/profiles.ts` exports `Profile` + `normalizeOwners` (clamp to [0,1] + renormalize) + `shareFor` (sums duplicate entries) + `splitMonthlyBurn` (per-profile base-currency burn map) + `reassignOnProfileDelete` (moves shares to a fallback) + `buildProfile`/`defaultProfile`/`normalizeProfiles`. Default profile id is `default` with `isDefault: true`. Subscriptions without `owners` are treated as 100% default profile.
- **Tests added**: 15 cases.
- **Verified**: Lib + tests green. UI wiring (ProfileSwitcher header, OwnersEditor slider, PerProfileBurn dashboard module) deferred.

## Feature 2 — P2P sync (WebRTC + QR) — Lib complete

- **Started**: 2026-05-14
- **Decisions**: `src/lib/peer-sync.ts` exposes `createHostSession`, `acceptHostOffer`, `completeHostHandshake`, `sendPayload`, `onPayload`, `closeSession`, plus `packDescription` / `unpackDescription` (base64url JSON wrapper for SDP). Default STUN: `stun:stun.l.google.com:19302`. ICE gathering waits up to 2s. `src/lib/qrcode.ts` implements binary-mode QR Code Model 2 at versions 1-10 with EC level L (Reed-Solomon over GF(256), polynomial 0x11d). Payloads over 271 bytes return a structured `{ ok: false, reason: "too-large" }` → UI falls back to copy-paste flow per the spec.
- **Tests added**: 6 peer-sync (packDescription round-trip + URL-safe output + malformed/missing-field rejection + default STUN) + 9 qrcode (each capacity boundary, structural SVG validity, module-count math, UTF-8 multibyte).
- **Verified**: Lib + tests green. UI wiring (PeerSyncFlow component with host/guest split + QR rendering + paste fallback) deferred.

## Feature 3 — Live calendar feed — Wired

- **Started**: 2026-05-14
- **Decisions**: Route handler at `src/app/s/[payload]/calendar.ics/route.ts`. `export const dynamic = "force-dynamic"`. Decodes the payload via the existing `sync.ts` helpers, strips `notes` from each subscription before passing to `serializeBurnRateIcs` (so the URL never leaks notes), and responds with `Content-Type: text/calendar; charset=utf-8` + `Cache-Control: public, max-age=300, s-maxage=300`. Malformed payloads return a 200 with a single-event ICS body so calendar clients don't error out.
- **UI**: `ShareAndData.tsx` gains a "Subscribe to live calendar" button under Sync. Generates a `BR3.` payload, builds a `webcal://<host>/s/<payload>/calendar.ics` URL, copies to clipboard with toast.
- **Tests added**: 3 (valid payload returns ICS with cache headers, garbage payload returns the invalid-feed body at 200, notes are stripped).
- **Verified**: Route + tests + UI wire-in green. Build confirms route exists.

## Feature 4 — Encrypted share links — Lib complete

- **Started**: 2026-05-14
- **Decisions**: `src/lib/crypto-share.ts` exports `encryptForShare`, `decryptFromShare`, `wrapSharePayload`, `unwrapSharePayload`, `isEncryptedSharePayload`. Envelope format: `ENC2.<base64url(salt|iv|ciphertext)>`. Share payload prefix: `BR5E.` (vs `BR5.` for plaintext). Uses the v3 crypto.ts PBKDF2 (250k iter SHA-256) → AES-GCM 256 primitives but with a distinct prefix and self-contained salt per share.
- **Tests added**: 10 cases (round-trip with right passphrase, wrong-passphrase rejection, tampered envelope rejection, IV uniqueness across calls, empty passphrase rejection, BR5E. prefix detection, wrap/unwrap round-trip, missing-prefix rejection).
- **Verified**: Lib + tests green. `/s/[payload]/page.tsx` extension (passphrase entry view + cooldown + generic OG for encrypted) deferred.

## Feature 5 — Decoy mode — Lib complete

- **Started**: 2026-05-14
- **Decisions**: `src/lib/decoy.ts` extends `VaultMeta` with an optional `decoy?: { verifier, salt, enabled }` block. `enableDecoy` requires the real passphrase + a different decoy passphrase. `resolveUnlock` returns `{ slot: "real" | "decoy", key }` — the unlock flow routes to the chosen slot transparently. `disableDecoy` wipes the decoy block. Demo data fixture (`DEMO_DECOY_DATA`) ships 6 plausible subs for the seed-from-template option. Storage slot namespacing is provided by `vault-registry.ts` (`namespacedKey(vaultId, "real"|"decoy", key)`).
- **Tests added**: 8 cases (passphrase verification before enabling, matching-passphrase rejection, real-vs-decoy unlock routing, fallback to throw on unknown, real-only path still works, disable).
- **Verified**: Lib + tests green. DecoySetup UI flow + lock-screen routing deferred.

## Feature 6 — Notification hub — Wired (settings panel)

- **Started**: 2026-05-14
- **Decisions**: `src/lib/notify.ts` exports `NotifySettings`, `defaultNotifySettings`, `scheduleAll`, `pruneFiredScheduled`, `ensurePermission`. Six channels: `renewal`, `trial-end`, `price-change`, `goal`, `discount-expiry`, `pending-cancel`. Pure scheduler emits events inside the 30-day horizon, respecting per-channel toggles and quiet-hours window. Each ScheduledNotification has a stable id so SW dedup works.
- **UI**: `NotificationSettings.tsx` Settings panel with master toggle gated by `ensurePermission()`, per-channel toggles, lead-time inputs, quiet-hours pickers, a "Test notification" button, and a status line that surfaces whether the browser supports `periodicSync`. Persistence: `burnrate.notify.v1`.
- **Tests added**: 9 cases.
- **Verified**: Lib + tests + Settings panel green. SW periodicSync registration + IDB scheduled-rows store still deferred (the page-level recompute + SW listener wiring).
